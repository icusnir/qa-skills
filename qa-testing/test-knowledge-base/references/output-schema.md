# Test Knowledge Base — Output Schema

The KB lives in a `knowledge-base/` directory (configurable via `--out` on the build script) and has one generated file plus five markdown layers.

```
knowledge-base/
├── catalog.json        # GENERATED — do not hand-edit
├── INDEX.md            # Tier 0 — domain → capability → coverage map
├── personas.md         # org × role personas and their usage
├── test-data.md        # accounts, payment methods, templates, fixtures, enums
├── dependencies.md     # cross-feature deps, feature flags, env/config, cleanup
└── domains/
    └── <domain>.md     # one per domain with real coverage
```

## `catalog.json` (generated by `scripts/build-catalog.mjs`)

```jsonc
{
  "generatedAt": "ISO timestamp",
  "repoRoot": "your-repo-name",
  "totals": {
    "featureFiles": 172, "scenarios": 537, "outlines": 219,
    "domains": 23, "personas": 29, "jiraTickets": 608
  },
  "domains": [                       // rolled up, sorted by scenario+outline count desc
    { "domain": "payment", "modules": ["ui"], "featureFiles": 27,
      "scenarios": 90, "outlines": 34, "jira": ["..."], "personas": ["..."], "tags": ["..."] }
  ],
  "personas": [                      // sorted by file usage desc
    { "persona": "MY_ORG SUPER_ADMIN", "org": "MY_ORG", "role": "SUPER_ADMIN",
      "files": 13, "domains": ["payment", "subscription", "..."] }
  ],
  "features": [                      // every feature file, sorted by path
    { "file": "ui-module/features/payment/cancelPaymentScenarios.feature",
      "module": "ui", "domain": "payment", "feature": "Payment cancellation scenarios ...",
      "featureTags": ["@ServiceOrg", "@Payments", "@PROJ-2577", "..."],
      "jira": ["PROJ-2577", "PROJ-2888"], "personas": ["MY_ORG SERVICE_MANAGER"],
      "scenarioCount": 5, "outlineCount": 1,
      "scenarios": [
        { "id": "<file>::<scenario title>", "scenario": "Successfully cancel ...",
          "type": "scenario", "tags": ["@PROJ-2888", "@Invoice"],
          "jira": ["PROJ-2888"], "personas": ["MY_ORG SERVICE_MANAGER"] }
      ] }
  ]
}
```

Scenario `id` = `<relative feature path>::<scenario title>` — stable across runs, used for diffing.

## `INDEX.md` (Tier 0 — load this first)

A single-screen map. For each domain, a row or short block:

- Domain name + module(s) + scenario/outline counts
- The capabilities it covers (grouped, business language — e.g. for `payment`: *create one-time / subscription / invoice payments, cancel, refund, filter, manage cadence plans*)
- Coverage depth signal (counts; note `@HappyFlow` vs `@Regression`)
- Pointer to `domains/<domain>.md` and the feature directory

Lead with the largest/most business-critical domains. This is the file consulted during a new PRD's gap analysis to decide which domain files to open.

## `personas.md`

Table of every persona (org × role), grounded in `catalog.personas`:

| Persona | Files | Domains | Used to exercise | Notes |
|---------|-------|---------|------------------|-------|

- Resolve org/role meaning from your environment handler / parameter-definition files where possible.
- Flag single-use personas (fragile) and high-traffic personas (regression-critical).

## `test-data.md`

What the suite reuses — so new test cases don't invent fixtures:
- **Accounts / payment-method tokens**: e.g. `INSUFFICIENT_FUNDS`, `US_BANK_ACCOUNT`, `CANCEL_ACCOUNT`, `MASTERCARD`, etc., with the domains/scenarios that use them and the behaviour they trigger.
- **Payload templates**: files under your test-data directories, grouped by purpose.
- **Upload fixtures**: files under your test-data files-for-upload directories.
- **Key enums**: the page-model enum files that encode product vocabulary (statuses, frequencies, etc.).

## `dependencies.md`

- **Cross-feature dependencies** — e.g. payment scenarios depend on a created loan/project; scenarios depend on upstream submissions.
- **Feature flags / gating** — any flag references or env-gated tags.
- **Environment constraints** — `@Stage`-only vs `@Devint01`-only scenarios and why.
- **Cleanup/teardown** — `@cleanup*` tags and what they tear down (state the test relies on being reset).
- **Config enums** — environment, org, site config that scenarios assume.

## `domains/<domain>.md`

Built from the catalog's `features[]` filtered to that domain. Sections:

1. **Summary** — what this domain covers, counts, modules.
2. **Capabilities** — grouped business-language list; under each, the scenarios that cover it with `@Jira` IDs and feature-file paths.
3. **Personas used** — from the domain rollup.
4. **Test data used** — tokens/templates seen in the domain's feature files.
5. **Dependencies** — setup the domain's scenarios require.
6. **Coverage notes** — happy-path-only areas, obvious gaps, `@only`/broken/debug exclusions.

Proportional: trivial domains (1 scenario) get a stub; exclude debug/broken pseudo-domains from coverage claims (note as non-coverage).
