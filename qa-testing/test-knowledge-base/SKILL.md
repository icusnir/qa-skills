# Test Knowledge Base — Build & Refresh

Produce and maintain a structured, queryable knowledge base of an existing Playwright/Cucumber (or Cucumber.js) test suite so that **test-strategy and test-case generation for new initiatives is coverage-aware**. The recurring question this answers: *"For this new PRD — what do we already test, what personas and data do we use, what does it depend on, and where are the gaps?"*

The KB has two layers with different jobs:
- **`catalog.json`** — the deterministic backbone, generated by a script. Never hand-edited. It is the source of truth for counts, scenario inventory, tags, Jira IDs, and personas. Regenerating it is cheap and exact, so it never drifts from the feature files.
- **Markdown layers** — the LLM-distilled, human-readable views (`INDEX.md`, `personas.md`, `test-data.md`, `dependencies.md`, `domains/*.md`). They carry the *business meaning* that raw scenario titles don't.

Output lives in a `knowledge-base/` directory at your project root (configurable via `--out` flag on the build script).

## When to use

- First-time build of the knowledge base.
- **Refresh** after feature files were added/changed/removed — re-run the script, diff, and re-distill only the domains that moved.
- Answering coverage questions directly ("do we already cover cancel-payment for ACH?").
- As an input to `test-strategist` Phase 5 (gap analysis) and to test-case generation.

## Inputs

- **The repo itself** — required. All `*.feature` files under your feature directories (e.g. `ui-module/features/`, `api-module/test/features/`), plus the supporting TypeScript/JavaScript that defines personas, test data, and configuration.
- **Previous `catalog.json`** — optional. When present, the script diffs against it and reports exactly which scenarios were added/removed/retagged and which domains need re-distilling.

### Source map — where each KB dimension comes from

| KB dimension | Source of truth in repo |
|--------------|-------------------------|
| Capabilities / expected behaviours | `*.feature` files: `Feature:`/`Scenario:` titles + Given/When/Then steps |
| Coverage tags & traceability | Gherkin tags: domain, Jira IDs, env (`@Stage`/`@Devint01`), type (`@E2E`/`@Regression`/`@HappyFlow`) |
| Personas | `ORG ROLE user logs into …` lines in Gherkin; resolved via your environment handler and parameter-definition files |
| Test data | Test-data directories (JSON payload templates, upload files), in-Gherkin tokens (e.g. `INSUFFICIENT_FUNDS`, `US_BANK_ACCOUNT`), enums under your page-model/enum files |
| Dependencies & configuration | Entity handlers, enums, environment tags, feature-flag references, cleanup tags (e.g. `@cleanup*`) |

## Workflow

### 1. Generate (or regenerate) the catalog — always first

Run the bundled parser from your project root. It has no dependencies (pure Node ≥18).

```bash
# First build (writes catalog.json to ./knowledge-base/)
node path/to/test-knowledge-base/scripts/build-catalog.mjs --root "$(pwd)"

# Refresh: diff against the committed catalog to see what changed
node path/to/test-knowledge-base/scripts/build-catalog.mjs --root "$(pwd)" \
  --diff ./knowledge-base/catalog.json

# Write to a custom output path
node path/to/test-knowledge-base/scripts/build-catalog.mjs --root "$(pwd)" \
  --out ./my-kb/catalog.json
```

The script writes `catalog.json` and prints totals. With `--diff` it prints, to stderr, the added/removed/retagged scenarios and the **list of domains to re-distill**. On a refresh, only touch the markdown for those domains — leave the rest unchanged.

> Tip: copy the existing `catalog.json` aside before regenerating if you want to diff against it after overwrite, or pass `--out` to write to a temp path first.

### 2. Distill the markdown layers from the catalog

Read `catalog.json` and produce the human-readable views described in `references/output-schema.md`. Ground every statement in the catalog and the feature files — **do not invent coverage that isn't there**. The cardinal rule: the feature files describe *what is tested*, not *what is correct*; never assert a behaviour is verified unless a scenario actually asserts it.

Produce/update:

1. **`INDEX.md`** (Tier 0) — one-screen map: every domain → its capabilities → coverage depth → pointers to the domain file and feature paths. This is the file `test-strategist` loads first.
2. **`personas.md`** — every org×role persona, how many files use it, which domains, and what it is used to exercise. Flag personas used in only one place (fragile) and the high-traffic ones (regression-critical).
3. **`test-data.md`** — accounts and payment-method tokens, payload templates, upload fixtures, and key enums; note which domains consume them. This is what test-case generation reuses instead of inventing fixtures.
4. **`dependencies.md`** — cross-feature dependencies, feature flags, environment constraints (`@Stage`-only, `@Devint01`-only), cleanup/teardown requirements, and config enums.
5. **`domains/<domain>.md`** — per domain: capabilities covered (grouped, in business language), the scenarios under each (with Jira IDs and feature-file paths), personas used, test data used, and dependencies. Use the per-domain `features[]` slice of the catalog.

Keep the markdown proportional — a 1-scenario domain gets a stub; a 100+ scenario domain gets a full capability breakdown. Drop pseudo-domains used for debugging from the narrative layers (note them as non-coverage).

### 3. Report

Summarize what was built/changed: totals, new domains, and — on a refresh — the diff (added/removed scenarios, re-distilled domains). If feature files referenced personas or test data that don't resolve in the supporting files, list them as findings, not silent omissions.

## Integration with `test-strategist`

When generating a strategy for a new PRD, `test-strategist` Phase 5 (gap analysis) should:
1. Load `knowledge-base/INDEX.md` to identify which existing domain(s) the PRD touches.
2. Load the relevant `domains/*.md` to list **regression candidates** (existing behaviour in the change surface) and **coverage gaps** (PRD requirements with no existing scenario).
3. Pull reusable **personas** and **test data** from `personas.md` / `test-data.md` so new scenarios reuse the established fixtures instead of inventing them.

This is the difference between "coverage is assumed greenfield" and a defensible gap analysis.

## Self-discipline

- **Catalog is generated, never hand-edited.** If a number looks wrong, fix the parser, don't patch the JSON.
- **Ground everything.** Every capability claim traces to a feature file and scenario. No fabricated coverage.
- **Distinguish tested from correct.** The KB records what scenarios *exercise*, not whether the expected outcomes are right.
- **Refresh is a diff, not a rewrite.** Re-distill only the domains the diff flags. Preserve curated prose elsewhere.
- **Proportional detail.** Big domains get depth; trivial ones get a line.
